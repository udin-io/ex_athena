# ADR 1: Use ReqLLM.ToolCall struct for assistant tool_call replay

## Status

Accepted

## Context

`ExAthena.Providers.ReqLLM.format_tool_calls/1` previously produced a flat `%{id, name, arguments}` map and attached it to `%ReqLLM.Message{tool_calls: ...}`. OpenAI hosted backends accepted the shape, but llama.cpp's strict parser rejected it on assistant tool_call replay (iteration 2+ of a tool-calling loop) with HTTP 500 `Failed to parse messages: Missing tool call type`.

The req_llm library (deps/req_llm/lib/req_llm/tool_call.ex) defines a canonical `%ReqLLM.ToolCall{id, type: "function", function: %{name, arguments: <json string>}}` struct that mirrors the OpenAI Chat Completions wire format. This shape is what every req_llm-supported backend, including llama.cpp, expects on the wire.

Internally ExAthena models tool-call arguments as Elixir maps for ergonomics, while the canonical struct expects arguments as a JSON-encoded string. The shape mismatch happens at the provider boundary.

## Decision

`format_tool_calls/1` constructs `%ReqLLM.ToolCall{}` via `ReqLLM.ToolCall.new/3`. Arguments are JSON-encoded at the boundary using the following rules:

- `nil` → `"{}"`
- binary (pre-encoded JSON string) → pass-through
- map → `Jason.encode!/1`

The `type: "function"` field is set by the library constructor and is not duplicated in caller code. Ids are auto-generated by the library when absent.

## Consequences

- llama.cpp and any other strict OpenAI-compatible backend now accept assistant tool_call replays past iteration 1.
- ExAthena's outbound payload aligns with req_llm's documented canonical model; future req_llm releases that tighten validation will not break us.
- Arguments are JSON-encoded once, only at the provider boundary; the rest of ExAthena continues to work with maps internally.
- Adds one dependency on `Jason.encode!/1` at this call site (already a transitive dep of the project).
- Behavior unchanged for OpenAI, Anthropic, Ollama, and other backends that already accepted the canonical shape.
